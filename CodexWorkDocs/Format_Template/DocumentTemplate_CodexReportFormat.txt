作成日時：報告作成日時を YYYY-MM-DD HH:MM:SS (Asia/Tokyo) 形式で記載する
更新日時：報告更新日時を YYYY-MM-DD HH:MM:SS (Asia/Tokyo) 形式で記載する

Codex側の作業完了時にユーザーへ回答する際は、以下のフォーマットに沿って回答すること。
記述ルールは UnityProject を作業ディレクトリとした相対パスで ../CodexWorkDocs/Development_Docs/CodexWorkGuide_CodexReportRules.txt を正とする。
通常の質問回答・相談・方針確認では、このフォーマットを使わず簡潔に回答すること。
該当しない条件付き項目は省略し、確認できていない項目は「未確認」と明記すること。

============================================================
【Commit Title】
============================================================
- 何をしたかを日本語で1行で記載する

【仕様の正の場所】
- ../CodexWorkDocs/Development_Docs（_DEPRECATED参照禁止。最新はファイル名そのままのもの）

【参照ドキュメント一覧】
- 仕様Docs：
  - ../CodexWorkDocs/Development_Docs/参照した仕様書ファイル名
- 補助資料（必要時）：
  - 参照した補助資料のリポジトリルート相対パス

【このコミットが関わる最終ゴール（大タスク）】
- ユーザーから提示された最終的な目標

【このコミットの作業と目的】
- 何を追加 / 変更して、何を可能にするのかを1〜3行で記載する
- 仕様上の根拠：どのDocのどの節に対応するかを簡潔に記載する

【作業範囲（今回触る / 触らない）】
- 触る：
  - 今回変更した対象
- 触らない（MVP外 / 今回対象外）：
  - 今回あえて触っていない対象

【変更ファイルと変更内容の一覧】
※パス表記はリポジトリルート起点の相対パスに従う。該当区分のみ記載する。
※Unityプロジェクト配下のファイルは UnityProject/Assets/...、UnityProject/Packages/...、UnityProject/ProjectSettings/... の形式で記載する。
- 追加：
  - UnityProject/Assets/...（何をどう追加したか）
- 更新：
  - UnityProject/Assets/...（何をどう直したか）
- 削除：
  - UnityProject/Assets/...（何を削除したか）

【未確定事項・TODO（あれば）】
- 未確定事項がある場合は、決めるべき内容を箇条書きで記載する
- TODOは、今回新規追加または内容変更したものだけを記載する

【作業確認報告（Codexが可能な範囲）】
- コンパイル：実行済み / 未実行 / 未確認。結果を簡潔に記載する
- 実行確認：実行済み / 未実行 / 未確認。結果を簡潔に記載する
- 仕様準拠：参照Docsとの矛盾有無、独自判断有無を簡潔に記載する
- 影響範囲：新規追加 / 更新した型や処理がどこから呼ばれる想定かを簡潔に記載する

【正式機能名関連の確認（採用プロジェクトのみ）】
- 正式機能名辞書の更新有無：正式機能名辞書を採用している場合のみ記載する
- 正式機能名に影響する判断：Yes / No。Yesの場合は根拠と反映先を簡潔に記載する

【開発状況ドキュメント関連の確認（採用プロジェクトのみ）】
- 開発状況ドキュメントの更新有無：開発状況ドキュメントを採用している場合のみ記載する
- 索引ドキュメントの更新有無：Yes / No。採用していない場合は省略する
- 対象個別ドキュメントの更新有無：Yes / No。採用していない場合は省略する
- 補助資料としての扱い：仕様の正としない補助資料を参照した場合のみ記載する

【健全性チェック（必要時）】
- 未使用コード / 未参照フィールドが大量に増えていない：Yes / No。Yesの場合は理由を記載する
- 既存責務を不必要に崩していない：Yes / No
- 不要なファイル追加 / 不要な大分類変更を行っていない：Yes / No
- プロジェクト固有の確認項目：
  - UnityProjectの既存フォルダ構成、命名、SerializeField / Tooltip方針から外れていない
  - 仕様書に未定義の挙動を恒久仕様として扱っていない

【備考（必要なら）】
- 必要な補足事項を記載する
