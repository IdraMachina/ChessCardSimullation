【基本方針】
このリポジトリでは、仕様書に定義された内容を優先して実装する。
仕様書に未定義の内容を、Codexの判断だけで恒久仕様として確定してはならない。
未定義の内容は、TODOコメント、暫定実装、または作業報告の確認事項として明記する。

【応答・コミット方針】
Codexからユーザーへの返答は日本語で行うこと。
コミットメッセージ、コミットタイトル、コミット本文、コミット内容の説明は日本語で記載すること。

【仕様参照の優先順位】
1. ユーザーから明示された最新の作業指示
2. リポジトリ内の仕様書・設計書（CodexWorkDocs/Development_Docs）
3. AGENTS.md
4. 既存コードの実装傾向

仕様書に未定義の挙動がある場合、Codexは推測で仕様を確定してはならない。
未定義の仕様は、暫定実装・TODO・確認事項として明記し、作業報告に記載すること。

【作業前確認】
作業開始前に、変更対象ファイル、関連仕様書、影響範囲を確認すること。
既存実装と仕様書が矛盾する場合、勝手に片方を正とせず、作業報告に「確認事項」として記載すること。

【変更範囲】
作業指示に含まれないファイルを変更する場合は、必要な理由を明記すること。
リファクタリング、命名変更、ファイル移動、責務分割は、作業目的に必要な場合のみ行うこと。

【未定義仕様の扱い】
仕様書に未定義の挙動がある場合、以下のいずれかで対応すること。

1. 実装を行わず、TODOコメントと作業報告に確認事項として残す。
2. テスト用・仮実装であることをコードコメントと作業報告に明記する。
3. 既存コードに同種の実装がある場合、その実装傾向に合わせる。ただし、その挙動を恒久仕様として確定した扱いにはせず、仕様書に未定義であること、既存実装に合わせた暫定対応であることを作業報告に明記する。

未定義仕様を恒久仕様として実装してはならない。

＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝

【命名規則】
変数名・引数名・一時変数名・関数名は、役割が分かる具体的な名前にすること。

禁止例：
- i, j, k, t, tmp, data, obj, manager, value など、役割が曖昧な名前
- Do(), Process(), Execute(), Handle() のように処理内容が分からない関数名

許可例：
- cardIndex
- laneIndex
- selectedUnit
- remainingMoveDistance
- targetBoardCell
- CalculateSiegeDamage()
- TryPlaceSchemeCard()
- ResolveUnitMovement()

＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝

【コメント方針】
コメントは、コードを読めば分かる内容を繰り返すために書かない。
コメントは、処理の意図、仕様上の理由、注意点、例外条件を説明するために書く。

public / protected / internal な関数：
- 原則として XML コメントを付ける。
- 役割、主な入力、戻り値、失敗条件がある場合はその条件を説明する。

private 関数：
- 処理が単純な場合はコメント不要。
- 分岐が多い、仕様由来の例外がある、副作用がある、複数段階の処理を行う場合は簡潔にコメントを付ける。

関数内コメント：
- 複雑な処理ブロックの前に、何を解決する処理かを1〜2行で書く。
- 1行ごとの実況コメントは禁止する。

＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝

【Unity Inspector公開フィールド】
Unity Inspectorで調整する値は、原則として private フィールドに [SerializeField] を付けて公開する。
public フィールドは、外部コードから直接書き換える必要がある場合を除き使用しない。

[SerializeField] を付けたフィールドには、原則として [Tooltip] を付ける。
Tooltipには、以下を簡潔に記載する。
- 何を制御する値か
- どの画面・機能に影響するか
- 単位、範囲、0やnullの意味がある場合はその意味

例：
[SerializeField]
[Tooltip("カードを盤面に配置できるか判定するときに参照する盤面管理クラス。未設定の場合は配置判定を行えない。")]
private BoardState boardState;

【作業後報告】
最終的な作業内容の報告には、リポジトリルートからの相対パスで CodexWorkDocs/Format_Template/DocumentTemplate_CodexReportFormat.txt を参照し、そのFormatに準じて報告する。
該当ファイルが見つからない場合は、以下の項目で報告する。

- 作業概要
- 変更ファイル
- 変更内容
- 仕様参照元
- 実行した確認
- 実行していない確認
- 未対応事項
- 確認が必要な事項
